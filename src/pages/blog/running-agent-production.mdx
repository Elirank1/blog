We deployed Eva — a personal AI assistant built on [OpenClaw](https://github.com/nicepkg/openclaw) — as an experiment. Not a product launch. An experiment to understand what it actually takes to operate an AI agent in the real world. Not in a demo, not in a sandbox — in production, connected to real messaging channels, handling real conversations.

We did this because we knew the community needed it. OpenClaw exploded in popularity, going from 9,000 to over 140,000 GitHub stars in days. Thousands of people were trying to set it up. We wanted to go first, hit the walls, and share what we found.

Within two weeks, our API bill exploded.

A single "hi" message was sending 169,000 tokens to the model. Every request cost $0.63. For a personal assistant that's supposed to save time and money, this was the opposite of the thesis.

Most people experience AI through chat interfaces — you type, it responds, you pay for what you use. Running an AI agent in production is something entirely different. The agent is always on. It processes heartbeats, monitors group chats, loads context, maintains sessions. The economics are not the economics of conversation. They're the economics of an always-on system.

This is what we learned.

## What We Thought We Were Building

The setup was straightforward. Eva ran inside a Docker container on macOS, connected to Telegram for direct messages and a team broadcasting group, and to WhatsApp for personal and group conversations. She used Claude Sonnet 4.5 as her primary model, with Anthropic's prompt caching enabled to keep costs reasonable.

The architecture was sound — Docker isolation, read-only filesystem, non-root user, manual confirmation for sensitive actions. We had followed the security playbook.

What we hadn't done was follow the economics playbook. Because for most people building agents, that playbook doesn't exist yet.

## What Actually Happened

Three signals appeared almost simultaneously:

- **The bill was climbing fast.** What should have been a modest personal assistant cost was heading toward hundreds of dollars in the first two weeks.
- **Rate limits kept hitting.** Anthropic's usage limits were being reached repeatedly. Eva would stop responding mid-conversation — not because she couldn't answer, but because the API was throttled.
- **The logs revealed the real problem.** Every request — even a one-word message — was consuming 169,000 input tokens. That's 85% of Claude's 200K context window, used up before Eva even started thinking about a response.

The cost wasn't coming from Eva being verbose. It was coming from the sheer volume of context the system was loading for every single interaction.

## Pulling the Thread

We mapped five areas to investigate, ranked by suspected cost impact. What followed felt less like debugging and more like peeling back layers of a system that was working exactly as configured — just not as intended.

### WhatsApp Was a Wide-Open Firehose

WhatsApp's DM policy wasn't set to "disabled." Group policy wasn't set to "allowlist." This meant Eva was silently processing messages from WhatsApp groups she had no business being in. Each message triggered a full API call with the entire context window attached.

This single misconfiguration accounted for an estimated 70% of the bill. Every casual message in every connected group was a $0.63 API call.

### The Cache Was a Mirage

Anthropic's prompt caching is supposed to reduce costs by reusing the system prompt across requests. Ours was invalidating constantly. Three factors were responsible:

- **Dynamic metadata.** OpenClaw injects timestamps and runtime info into the system prompt. These change every message. One changed byte invalidates the entire cache.
- **Short TTL.** The cache TTL was set to 5 minutes. For a personal assistant you check a few times a day, every interaction was a cold start.
- **Tool call multiplication.** Each tool call resends the full context. One message triggering 5 tool calls means 5x the context sent.

The caching system wasn't broken. It was working exactly as designed — for a use case that wasn't ours.

### 48 Zombie Sessions

OpenClaw had accumulated 48 active sessions, mostly from WhatsApp groups. Each session maintained its own context window. We ran the cleanup command. The sessions disappeared from the dashboard. Then they came back.

The cleanup command only clears metadata. The actual transcript files — JSONL logs of every conversation — survived on disk and were reloaded on restart. The ghosts kept coming back because we were only clearing the index, not the data.

### A 2.1MB Transcript Nobody Knew About

The main session's transcript had grown to 2.1 megabytes. Nine hundred lines of conversation history, sent in full with every single request. This file alone was responsible for the majority of the 169K tokens.

Context pruning was configured. It just wasn't being applied to existing transcripts. The settings were forward-looking. The damage was already done.

### The Wrong Pricing Model

Claude Sonnet 4.5 charges $3.75 per million tokens for prompt cache writes, on top of $3 per million for regular input. For an agent that sends a large, mostly-static system prompt with every message, this pricing model is punishing.

Gemini 2.5 Pro charges a flat $1.25 per million input tokens. No separate caching surcharges. For our access pattern — frequent, context-heavy, cache-hostile — the economics were dramatically different.

## What We Refused to Sacrifice

Before touching anything, we defined what couldn't break:

- **Eva's identity.** Her personality, communication style, and behavioral guidelines — defined in SOUL.md — had to remain intact. You don't fix costs by lobotomizing your agent.
- **Security posture.** Docker isolation, Telegram allowlists, manual confirmation for sensitive actions. None of this was negotiable.
- **Operational capability.** Broadcasting to the team group, managing memory, running periodic checks. Eva needed to keep doing her job.
- **Cost sustainability.** The fix had to make Eva cheap enough to use daily without thinking about it.

## What We Changed

**Locked down WhatsApp.** Set dmPolicy to "disabled" and groupPolicy to "allowlist." Eva stopped processing messages from unauthorized sources immediately.

**Extended cache lifetime.** Cache TTL moved from 5 minutes to 1 hour. Context pruning set to "cache-ttl" mode. Fewer cold starts, fewer full rewrites.

**Killed the zombie sessions — for real.** Deleted all 48 sessions and their JSONL transcript files. Not just the metadata — the actual data. The transcripts that kept resurrecting on restart were gone.

**Trimmed the system prompt.** Deleted BOOTSTRAP.md — a first-run file the agent should have removed after setup but never did. Removed a duplicate SOUL.md that was being loaded alongside the real one. Cut redundant sections from AGENTS.md that OpenClaw's built-in instructions already covered. Net savings: ~3,750 tokens per request.

**Switched to Gemini 2.5 Pro.** Moved from Claude Sonnet 4.5 to Gemini 2.5 Pro. Flat input pricing, no caching surcharges. The model config had to be set as an object with a "primary" key — a plain string silently failed. No error, no warning. The agent just kept using the old model.

**Closed the Telegram security gap.** Added sender allowlist to the Telegram broadcasting group. Security audit: zero criticals.

## What This Teaches About Production Agents

**Measure tokens before features.** The first metric to track in a production agent is input token count, not response quality. If your agent sends 169K tokens for a "hi," nothing else matters until you fix that. Instrument your token flow from day one. Everything downstream depends on it.

**Open inputs create open cost surfaces.** Every messaging channel your agent connects to is a potential cost surface. WhatsApp bridges forward everything. Telegram groups can be noisy. Each message triggers the full context pipeline. Lock channels to allowlists before connecting, not after the bill arrives.

**Session cleanup is not what you think.** In OpenClaw, the cleanup command clears metadata but leaves transcript files on disk. On restart, the sessions reappear. Your framework probably has a similar gap. Don't assume cleanup means deletion. Verify what's actually on disk.

**Dynamic prompts kill caching.** If your system prompt includes timestamps, runtime metadata, or anything that changes per-request, you're invalidating the cache every time. One changed byte triggers a full rewrite. Separate what's static from what's dynamic, or accept that caching won't help you.

**Silent config failures are the norm.** Our model switch silently failed because the config expected an object and received a string. No error. No warning. The agent kept running on the expensive model. Always verify changes by inspecting runtime behavior, not just config files.

**Choose pricing structure, not just capability.** Claude's prompt caching charges $3.75/M tokens for cache writes. Gemini's flat rate is $1.25/M input. For an always-on agent with a heavy system prompt, the pricing structure matters more than the per-token price. Match your model's economics to your agent's access pattern.

**Workspace files are your hidden context budget.** Every markdown file in the agent's workspace gets injected into every request. A forgotten setup file, a duplicate personality doc, an oversized instruction set — they compound silently. Treat workspace files like production code: review, trim, delete what you don't need.

**Security and cost are the same problem.** An open WhatsApp policy is both a security vulnerability and a cost explosion. A missing Telegram allowlist is both a command injection risk and an invitation for unauthorized API consumption. When we hardened security, costs dropped. When we cut costs, the attack surface shrank. They're not two problems — they're one problem viewed from different angles.

## Why We Did This

I built Deeplica because I believe the digital world should serve people, not consume them. Eva isn't a product we deployed — she's an experiment we ran for the community. We knew OpenClaw would have rough edges. We knew the people rushing to set it up would hit walls that aren't documented anywhere. So we went first.

The irony was hard to miss. I was building an "attention guardian" — an agent designed to absorb operational noise so I could focus on what matters. Instead, it was generating the most expensive kind of noise: silent, invisible, and compounding.

What I didn't expect was how different agent economics are from chat economics. In a chat window, you pay per conversation. In production, you pay per system heartbeat, per group message, per tool call, per session that forgot to die. The meter is always running, and if you're not watching the token flow, nobody is.

The fixes themselves were straightforward. Lock down your channels. Match your model to your access pattern. Clean up your context. Monitor your tokens. None of it is glamorous. But it's the difference between an agent you can actually live with and one that lives on your credit card.

Eva is lean now. Fourteen thousand tokens per request. Two cents per message. Zero security criticals. She's doing what she was always meant to do — working quietly in the background so I can be present in the foreground.

Running AI agents is not a prompting problem. It's an architecture and economics problem. The sooner the industry internalizes that, the sooner agents will actually work for the people they're supposed to serve.
